1. Database Structure

-- -----------------------------------------------------
-- 1. Main sensor data table (simulator writes here)
-- -----------------------------------------------------
CREATE TABLE SensorReadings (
    RowGUID CHAR(36) PRIMARY KEY,
    SensorTimestamp DATETIME2 NOT NULL,   -- time when the row was dumped (set by simulator)
    TagID INT NOT NULL,
    SensorDate DATETIME2 NOT NULL,        -- random date generated by simulator (sensor's original timestamp)
    Value FLOAT NOT NULL,
    Quality VARCHAR(20) DEFAULT 'Good'
);
GO

-- -----------------------------------------------------
-- 2. Outage event log (records every offline/online transition)
-- -----------------------------------------------------
CREATE TABLE OutageLog (
    OutageID INT IDENTITY PRIMARY KEY,
    ServerName VARCHAR(50) NOT NULL,      -- 'PRIMARY' or 'SECONDARY'
    ServerEvents VARCHAR(50) NOT NULL,    -- 'offline' or 'online'
    EventDateTime DATETIME2 NULL,         -- when the event was detected
    SyncStatus VARCHAR(20) DEFAULT 'Pending',
    Notes VARCHAR(500) NULL
);
GO

-- -----------------------------------------------------
-- 3. Sync audit table (stores rows missed by remote server)
-- -----------------------------------------------------
CREATE TABLE SyncRecordsStatus (
    RowGUID CHAR(36) PRIMARY KEY,
    SensorTimestamp DATETIME2 NOT NULL,
    TagID INT NOT NULL,
    SensorDate DATETIME2 NOT NULL,
    Value FLOAT NOT NULL,
    Quality VARCHAR(20) DEFAULT 'Good',
    SyncSuccess BIT DEFAULT 0             -- 0 = not yet synced to remote, 1 = synced
);
GO

5. How the Version 3 Flow Works (Step‑by‑Step)

Normal operation (both online):
Every 30 seconds, the WS script checks both servers.
Both are online → log "SECONDARY is ONLINE" and "PRIMARY is ONLINE".
No action needed.

Outage begins (SECONDARY goes down at 10:30:05):
At 10:30:30, health check fails → secondary_status becomes -1 (offline).
outage_start is recorded as 10:30:30.
OutageLog row inserted on PRIMARY: ServerName='SECONDARY', ServerEvents='offline', EventDateTime=10:30:30.

During the outage:
Every 30 seconds, capture_rows_for_sync() runs.
It copies any new rows from PRIMARY's SensorReadings (where SensorTimestamp >= 10:30:30) into PRIMARY's SyncRecordsStatus with SyncSuccess=0.
Additional OutageLog rows may be inserted each cycle (since the server is still offline), each with a new EventDateTime.
(If you want only one offline entry, we can suppress duplicates – but multiple entries show the duration.)

Outage ends (SECONDARY comes back at 12:44:55):
At 12:45:00, health check succeeds → transition detected.
OutageLog row inserted on PRIMARY: ServerEvents='online', EventDateTime=12:45:00.

sync_to_remote() is called:
Reads all rows from PRIMARY's SyncRecordsStatus where SyncSuccess=0.
Pushes them to SECONDARY's SensorReadings (using RowGUID to skip duplicates).
Updates those rows in PRIMARY's SyncRecordsStatus to SyncSuccess=1.
Logs the number of synced rows.

Result:
SECONDARY's SensorReadings now has all rows dumped during the outage.
PRIMARY's SyncRecordsStatus holds a permanent audit of exactly which rows were missed and later synced.
OutageLog contains a timeline of the entire event.
